@AGENTS.md

## Project

Bhagwandas Traders — a path-based, multi-tenant Next.js App Router grocery SaaS platform. Each store lives under `/[store]/...`, with tenant isolation enforced in the Prisma schema and in `proxy.ts`. Everything is backed by a real Postgres database via Prisma: stores, users/auth (with email verification), the product catalog, file uploads, transactional email (with an `EmailLog` audit trail), and the full order lifecycle (checkout → verification → packing → rider assignment → delivery). The only purely local/ephemeral things are in-progress UI state with no reason to be persisted — e.g. which row is selected in a table, a staff packing checklist's checked/flagged state, a delivery rider's compliance checklist before confirming drop-off.

## Commands

```bash
npm run dev         # start dev server at http://localhost:3000 (Turbopack is the Next 16 default)
npm run build       # production build
npm run start       # run the production build
npm run lint        # ESLint (flat config: eslint-config-next core-web-vitals + typescript)
npm run db:migrate  # apply Prisma migrations to DATABASE_URL (also auto-seeds)
npm run db:seed     # (re-)seed demo stores/users/products/orders from prisma/seed.ts
```

There is no test suite/framework configured. Always run `npm run lint` and `npm run build` before considering a change done (per README's validation step) — `build` is the only type-check pass since `lint` alone won't catch TS errors.

Local setup requires `.env` (see `.env.example`): `DATABASE_URL` (+ `DIRECT_URL` for migrations), the `SMTP_*` variables for transactional email, and `APP_BASE_URL` (absolute origin used to build links embedded in emails). The three `SUPABASE_*` realtime vars are optional — without them, live dashboard updates degrade to 30s polling. `postinstall` runs `prisma generate`. Seeded demo logins all use password `password123` (e.g. `admin_user` / `staff_user` / `delivery_user` on the `bhagwandas-traders` store).

## Architecture

**Route files are thin; components hold everything.** Every file under `app/**/page.tsx` does nothing but fetch data and render one component from `components/`, named `<route>-page.tsx` (e.g. `app/[store]/admin/orders/page.tsx` renders `components/admin-orders-page.tsx`). When asked to change a page's behavior or markup, go straight to its `components/*.tsx` file, not the route file. (Exception: `app/[store]/verify-email/page.tsx` is self-contained — it consumes the token and renders inline.)

Every `app/**/page.tsx` is an `async` Server Component that fetches real data via Prisma (and, for protected routes, `verifySession()`/`requireRole()`) and passes it down as props — the `"use client"` page components underneath own transient UI state via `useState`/`useMemo` (search filters, selected row, drawer/modal open state), but business data always comes from the database. Mutations go through `"use server"` Server Actions in `app/actions/*.ts` (`auth.ts`, `orders.ts`, `inventory.ts`, `staff.ts`, `store-settings.ts`, `store.ts`), each calling `revalidatePath()` so the relevant Server Component re-fetches after a write. Form-bound actions follow the `useActionState` shape `(state, formData) => Promise<State>` returning `{ error }` or `{ fieldErrors }`.

**Real routes live only under `app/[store]/...`, plus `/signup` and `/api`.** The top-level `app/admin`, `app/checkout`, `app/delivery`, `app/login`, `app/order`, `app/register`, `app/staff` directories are empty leftovers from the pre-multi-tenant layout — don't put new routes there. Dynamic routes use the async `params` convention — e.g. `app/[store]/order/[id]/page.tsx` awaits `params: Promise<{ store: string; id: string }>`.

**Multi-tenant routing**: tenant resolution is path-based, not subdomain-based; the tenant URL segment is always the first path segment (`/[store]/...`). `app/[store]/layout.tsx` is the existence gate (404 for unknown slugs) via the cached `getStoreBySlug` in `lib/store.ts`. `proxy.ts` (Middleware) stays database-free: it reads the plain-JSON session cookie, skips slugs in `RESERVED_STORE_SLUGS` (`lib/reserved-slugs.ts`: `signup`, `api`), redirects logged-in users away from `/login`/`/register`, and gates `admin`/`staff`/`delivery` sections by role plus `checkout` and `order/[id]` by login — always comparing the URL store slug against `session.storeSlug`. Because Middleware must never transitively import Prisma/bcrypt, the `Role` type and `roleHome()` (per-role landing page) live in the zero-dependency `lib/roles.ts`. Public onboarding is `/signup` (`app/actions/store.ts`), which creates a `Store` + first admin user in one transaction and derives a unique 3-letter `codePrefix` from the store name.

**Database**: `prisma/schema.prisma` defines `Store`, `User`, `Product`, `Order`/`OrderItem`, and `EmailLog`. `User`, `Product`, `Order`, and `EmailLog` each carry a required `storeId` foreign key and are indexed by it; `User` has compound uniques on `[storeId, username]` and `[storeId, email]`. `Store` owns settings fields (`openingTime`, `closingTime`, `hourlyCapacity`, editable at `/[store]/admin/settings`) plus `slug`, `codePrefix`, and an optional `upiId` (used to render a server-generated UPI QR via the `qrcode` package in `app/[store]/checkout/page.tsx`). All money is integer rupees (`Int` columns, `formatCurrency` in `lib/format.ts` renders `en-IN` INR).

**Auth & sessions**: `lib/users.ts` scopes `findUserByCredentials`, `findUserByUsername`, `findUserByEmail`, `registerCustomer`, and `createStaffUser` by `storeId`, hashing with `bcryptjs`, and re-exports `Role` (from `lib/roles.ts`) and Prisma's `User` as `AppUser`. `lib/session.ts` stores the session as **plain unencrypted JSON** in an httpOnly cookie (`id`, `username`, `name`, `role`, `storeId`, `storeSlug`) — a deliberate lightweight choice, no rotation/revocation; "remember me" sets a 30-day `maxAge`. `verifySession()` (cached, redirects to `/` if absent) and `requireRole(...roles)` are the guards used by pages and actions. Registration and signup issue a 24-hour `emailVerifyToken`; `/[store]/verify-email?token=...` confirms it. Verification is non-blocking — unverified users can still log in and order.

**Order lifecycle** (`app/actions/orders.ts`): `placeOrder` re-prices the cart server-side, enforces a ₹300 minimum, requires a payment-proof upload for UPI (skipped for COD via `paymentMethod`), then in one transaction creates the order, decrements stock with a `stock: { gte: quantity }` guard (turning races into a friendly out-of-stock error), and assigns `displayId` = `{codePrefix}-{YYMMDD}-{100 + orderNumber}`. Status flow: `pending_verification` →(admin `verifyOrder`)→ `packing` →(staff `dispatchOrder`)→ `out_for_delivery` →(delivery `completeDelivery`)→ `delivered`; the `verified` enum value exists but is not used as a resting state. Riders: admin `assignRider` sets `Order.riderId`, delivery `acceptPickup` stamps `acceptedAt`; staff can `toggleBreak` (`User.onBreak`). Human-readable status labels live in `lib/order-status.ts`.

**Cart & pricing**: there is no client cart store — the catalog passes `?items=productId:qty,...` to `/checkout`. `lib/cart.ts` is pure and client-safe (`parseCartParam`, `cartParamFromLines`, `computeComboDiscount` — flat ₹20 per combo-eligible line with qty ≥ 3); `lib/pricing.ts` (`priceCart`, server-only) recomputes totals from live `Product` rows so nothing is trusted from the client.

**Email**: `lib/mail.ts` wraps a singleton nodemailer transporter; `sendMail` writes every attempt (sent or failed) to `EmailLog` and never propagates send failures beyond a console error. HTML templates are plain functions in `lib/emails/*` returning `{ subject, html }`, keyed by the `EmailType` enum (registration welcome, order confirmation, out-for-delivery, delivered). Links in emails must be built with `buildAppUrl()` (uses `APP_BASE_URL`).

**Realtime** (`lib/realtime.ts` + `lib/use-store-events.ts`): order mutations in `app/actions/orders.ts` call `publishStoreEvent(storeId, ...)`, which POSTs a minimal `{ orderId }` payload to Supabase Realtime's HTTP broadcast endpoint on the per-tenant channel from `lib/store-channel.ts` (`store:{storeId}`) — never throws, 3s timeout, no-op when the `SUPABASE_*` env vars are unset. Clients (the three operations dashboards via the `useStoreEvents(store.id)` hook; the customer order-tracking page via the `<StoreEventsRefresher>` null component, since it's a Server Component) subscribe over WebSocket and respond with a debounced `router.refresh()`, so fresh data still flows through the existing session-scoped Server Component fetches; without env config they fall back to 30s polling. These files are deliberately the **only** Supabase-SDK touchpoints in the app — never call `@supabase/supabase-js` from anywhere else, so swapping realtime providers stays a two-file change. Events are a doorbell, not a data channel: don't put order details in payloads.

**File uploads** (`lib/storage.ts`): public assets (product images, staff avatars) go to `public/uploads/<storeId>/<category>/` via `savePublicUpload` and are served statically; sensitive files (payment-proof screenshots) go to `private-uploads/<storeId>/...` via `savePrivateUpload` and are served only through `app/api/files/[storeId]/[...path]/route.ts`, which requires a session in the same store with role admin/staff/delivery.

**Shells**: `components/dashboard-shell.tsx` (`DashboardShell`) is the sidebar nav + header shell for all operations surfaces (admin orders/inventory/users/settings, staff packing, delivery dashboard + complete) — nav items are filtered by a `currentRole` prop, highlighting is driven by a `currentPath` prop, and its logout form binds `logout(storeSlug)`. `components/storefront-shell.tsx` (`StorefrontShell`) exists but is **not actually used** by the catalog/checkout/order-tracking pages — each has its own inline header (a pre-existing inconsistency). `components/account-menu.tsx` requires `storeSlug` so login/logout links stay tenant-scoped. `lib/csv-export.ts` (`downloadCsv`) is a browser-only helper used by dashboard pages for CSV export.

**Styling**: Tailwind v4 via `@tailwindcss/postcss` (no separate Tailwind config build step beyond `postcss.config.mjs`). Brand design tokens (colors, shadow, radius) are custom theme extensions in `tailwind.config.ts` under the `brand.*` namespace (`brand.green`, `brand.orange`, `brand.panel`, `brand.sidebar`, etc.) — use these rather than introducing ad hoc colors. `lucide-react` is the primary icon set.

**Design references**: `UI Mock UP/` contains one HTML mockup per screen (login, storefront, checkout, order confirmation, orders queue, inventory, packing station, delivery manifest/confirmation, rider-assignment modal, user management). When implementing or revising a page's visual design, check the matching mockup first before inventing new UI.

**This repo pins a bleeding-edge Next.js version** (16.2.10, App Router only) — per `AGENTS.md`, do not assume training-data knowledge of Next.js APIs/conventions applies. Check `node_modules/next/dist/docs/` (organized into `01-app`, `02-pages`, `03-architecture`, `04-community`) before using any Next.js API you're not certain about in this version, and follow any deprecation notices found there. Example: Next 16 renamed the Middleware convention — the file is `proxy.ts` with a default-exported `proxy` function, not `middleware.ts` (see `03-file-conventions/proxy.md`).
